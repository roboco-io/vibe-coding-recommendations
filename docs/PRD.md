# 바이브 코딩 표준 가이드 PRD

## 1. 개요

### 1.1 프로젝트명
**Vibe Coding Recommendations** - 바이브 코딩 표준 가이드

### 1.2 목적
AI 기반 코딩 도구(Claude Code, Cursor, GitHub Copilot 등)를 활용한 "바이브 코딩" 방법론에 대한 표준 가이드를 제공한다. 개발자들이 AI 도구를 효과적으로 활용하여 생산성을 높이고, 일관된 품질의 코드를 작성할 수 있도록 돕는다.

### 1.3 배경
- AI 코딩 도구의 급속한 발전으로 개발 방식이 변화하고 있음
- "바이브 코딩"이라는 새로운 개발 패러다임 등장
- 체계적인 가이드 부재로 인한 품질 편차 발생
- 조직 내 표준화된 AI 도구 활용 방법론 필요

---

## 2. 목표

### 2.1 핵심 목표
1. **표준화**: 바이브 코딩에 대한 일관된 가이드라인 제공
2. **교육**: 세션 발표를 통한 지식 전파
3. **접근성**: GitHub Pages를 통한 문서 공개 배포

### 2.2 성공 지표
- 가이드 문서 완성도 (핵심 주제 커버율)
- 발표 슬라이드 품질 (Marp 형식)
- GitHub Pages 배포 완료 및 접근 가능성

### 4.2 가이드 문서 구성

단일 파일(`guide.md`) 내에 다음 섹션들을 포함한다. 콘텐츠가 충분히 확장되면 개별 파일로 분리할 수 있다.

- 들어가며
  - 이 문서는 2024년 11월부터 2026년 1월까지 작성자의 개인적인 바이브 코딩 경험에 의해 작성되었음.
  - 자신만의 바이브 코딩 프로세스를 가진 사람 보다는 이제 막 바이브 코딩을 시작하려는 개발자들에게 도움이 되고자 함.
  - 작성자는 95년부터 30년간 개발자로 일해왔음(https://github.com/serithemage).
- 이 문서의 목적
  - 빠르게 변화하는 AI 코딩 도구 환경에서 개발자들이 일관된 품질과 생산성을 유지할 수 있도록 돕기 위함.
  - 이 문서 자체도 주기적으로 업데이트 될 예정이며, 특정 버전에 대한 참조는 Git 태그를 통해 관리된다.
- 바이브 코딩의 정의
  - 현 시점의 바이브 코딩은 단순히 자연어로 코드를 생성하는것이 아니라, AI 도구를 최대한 활용하여 개발 생산성을 극대화하는 방법론을 총칭한다.
- 핵심 원칙
  - 바이브 코딩은 기존의 소프트웨어 개발과 다르지 않다. 기존 소프트웨어 개발의 모범 사례와 안티패턴이 공유된다. 솔리드원칙, 클린코드, TDD, 디자인패턴, 애자일, 데브옵스 등 기존의 소프트웨어 개발 방법론이 여전히 유효하다. 하지만 앞으로도 그러리라는 보장은 없다.
  - 다만 바이브 코딩은 기존 소프트웨어 개발, 특히 모던 소프트웨어 엔지니어링(https://www.davefarley.net/?p=352)에 필요한 여러 활동들을 적은 비용으로 빠르게 수행할 수 있게 해준다.
  - AI를 믿지 말라! 사람도 믿지 말라! 신뢰 할 수 있는 프로세스를 만들어라!
  - 바이브 코딩에 있어서 AI나 도구만으로 낼 수 있는 생산성은 고정된 수치에 가까우며 한계가 있다. 개발자 자신의 역량과 올바른 도구, 프로세스가 결합되었을때 비로소 생산성이 증폭된다. 이를 공식으로 표현하면 다음과 같다.
    - 바이브 코딩 생산성 = 개발자 역량 x AI도구와 프로세스의 생산성
      - 개발자 역량이 없이는 AI 도구의 생산성도 제한적이다. 바이브 코딩의 생산성은 증가되는 것이 아니라 증폭되는 것이다.
      - 개발자 역량에서 코드 작성능력이 차지하는 비중이 빠르게 줄고 있다. 코더로서의 능력보다는 문제 해결능력, 도메인 지식, 아키텍처 설계능력, 프로세스 설계능력, 보안에 대한 지식이 차지하는 비중이 커지고 있다. 우리는 프로그래밍 언어를 다루지 않는 소프트웨어 개발자의 시대를 준비해야 한다.
- 도구
  - 현시점에서는 종합젹인 능력에 있어서 클로드(모델)/클로드 코드(개발도구)가 가장 우수하다. 클로드/클로드 코드 발전 속도 또한 빠르기 때문에 경쟁자들과 6개월 정도의 차이를 유지하고 있다. 하지만 시장은 치열하며 경쟁자들도 만만치 않기 때문에 주기적인 평가가 필요하다.
  - YOLO 모드를 사용한다. 사람에 의한 개별 리뷰보다는 자동화된 테스트, 리뷰, 백업 루틴을 통해 신뢰도를 확보한다.
  - CI/CD, IaC, 모니터링 등 데브옵스 툴체인과의 통합을 적극 활용한다.
  - 프로그래밍 언어
    - 백엔드에는 GO언어, 프론트엔드는 타입스크립트/리액트를 주로 사용한다.
      - 특히 go언어는 컴파일 속도, 정적타입, 간결한 문법, 뛰어난 성능 덕분에 바이브 코딩에 매우 적합하다.
    - 멀티플랫폼 개발에는 Flutter/Dart를 사용한다.
    - Makefile을 이용해서 프로젝트 구조, 테스트 스캐폴딩, 린터 설정등을 템플릿화 하여 AI에게도 동일한 지침을 제공한다.
  - MCP보다는 CLI를 선호. 이유는 스트립트를 통한 자동화가 가능하고, 토큰 사용량이 적기 때문임.
  - MCP는 최소한만 사용하는데, 조사 목적의 퍼플렉시티 MCP와 최신 문서 참조를 위한 Context7 MCP만을 사용한다.
- 프롬프팅
  - 요청 뿐만 아니라 의도를 전달하라. `XY 문제`는 문제 해결에 있어서 중요한 개념이며, 바이브 코딩에 있어서는 특히 그러하다.
  - AI에게 질문을 허락하라. 앤트로픽의 엔지니어가 제안한 방법론(https://x.com/trq212/status/2005315275026260309)은 매우 유용하다. 
  - 반복적으로 발생하는 프롬프트를 어떻게 관리할 것인가? 클로드 코드에는 에이전트, 커맨드, 훅, 스킬 그릭고 이들을 손쉽게 관리할 수 있는 플러그인 기능을 제공한다. 이러한 기능들을 적극 활용하라.
- 워크플로우
  - 요구사항 분석 > 테크 스택 및 아키텍처 설계 > 계획 수립 > 구현/검증/배포 루프
  - 정기적으로 전체 리뷰 및 이에 따른 리팩토링을 수행
- 품질 관리
  - 확인이 가능한 단위로 나누어서 구현과 테스트를 동시에 진행.
  - 자동화된 테스트와 CI/CD 파이프라인 구축
    - 깃이나 클로드 코드의 훅 기능을 활용해서 UT, 린트, 보안 스캔등을 촘촘하게 구성
- 보안
  - ZDR(Zero Data Retention) 모드 ON 필수
  - 자동화된 보안 리뷰를 파이프라인속에 포함
- 협업
  - 개발자 1명이 소화할 수 있는 영역이 넓어짐에 따라 오너십에 대한 재 정의가 필요
  - 협업 대상은 사람 뿐만 아니라 AI 도구도 포함된다
    - 컨텍스트를 어떻게 효과적으로 관리/공유할 것인가?
      - 문서화, 이슈트래커, 위키,  
  - 프로세스, 프롬프트는 팀 전체가 공유 (ex: 클로드 플러그인)
  - 사내 바이브 코딩 커뮤니티를 만들어서 모범 사례를 적극적으로 공유
  - 지식공유에 페어 프로그래밍이나 몹 프로그래밍을 적극 활용
- 팁 & 유용한 리소스
  - 수시로 업데이트 되는 앤트로픽의 공식문서와 튜토리얼, 기술 블로그를 주시하라.
  - 래딧의 앤트로픽 포럼(https://www.reddit.com/r/Anthropic/)
  - 긱뉴스 (https://news.hada.io/)
  - 플라스크 개발자인 아르민 로나허나 클로드 코드의 개발자인 Boris Cherny의 트위터, 유튜브, 블로그
