# 바이브 코딩 가이드

## 들어가며
이 가이드는 2024년 11월부터 2026년 1월까지 축적한 작성자의 바이브 코딩 실험과 프로젝트 기록을 토대로 정리되었다. 이미 자신만의 프로세스를 갖춘 전문가라기보다 이제 막 바이브 코딩을 시도하려는 개발자에게 길잡이가 되도록 구성했으며, 1995년부터 30년 가까이 개발자로 활동해 온 [작성자](https://github.com/serithemage)의 시행착오와 학습결과에 기반하고 있다.

## 1. 이 문서의 목적
빠르게 변화하는 AI 코딩 도구 환경에서 개발팀이 일관된 품질과 생산성을 유지하도록 돕는 것이 이 문서의 존재 이유다. 가이드는 단순한 참고자료가 아니라 프로젝트에서 반복적으로 학습한 통찰을 축적하는 살아 있는 표준으로 취급한다. 주요 변경은 Git 태그로 버전 관리하여 모든 팀원이 동일한 지침을 바라보게 하고, 실제 프로젝트 피드백을 반영해 주기적으로 검증·개선한다.

## 2. 바이브 코딩의 정의
바이브 코딩은 자연어로 코드를 생성하는 기술에만 국한되지 않는다. AI 도구 전반을 활용해 설계, 구현, 검증, 문서화, 운영까지 모든 소프트웨어 활동을 가속화하고, 개발자가 문제 정의와 의사결정에 더 많은 에너지를 투입하도록 만드는 전방위적 방법론이다. 핵심 목표는 코드 작성 속도가 아니라 문제 정의와 가설 검증 루프를 짧게 가져가는 데 있다.

## 3. 핵심 원칙
바이브 코딩은 기존 소프트웨어 공학의 연속선 위에 놓여 있다. SOLID, TDD, 디자인 패턴, 애자일, 데브옵스 같은 익숙한 방법론은 여전히 유효하며, AI는 이러한 실천을 더 적은 비용으로 더 자주 반복하게 해 주는 촉매제다. Dave Farley가 제시한 모던 소프트웨어 엔지니어링 활동(https://www.davefarley.net/?p=352)을 예로 들면, AI는 실험과 자동화를 통해 그 활동을 고속으로 수행하도록 뒷받침한다. 무엇보다 "AI를 믿지 말고, 사람도 믿지 말라"는 메시지처럼 결과를 검증할 수 있는 프로세스를 먼저 설계해야 한다. 자동화된 테스트, 리뷰, 백업 루틴이 그 역할을 담당한다. 생산성을 바라볼 때는 `바이브 코딩 생산성 = 개발자 역량 × AI 도구와 프로세스의 생산성`이라는 공식을 기억한다. 문제 해결력, 도메인 이해, 아키텍처·프로세스·보안 역량이 미약하면 AI가 제공하는 생산성도 한계가 있고, 역량이 높을수록 AI가 곱셈 효과를 낸다. 결국 프로그래밍 언어를 직접 다루지 않는 시대를 대비해 문제 정의와 시스템 사고 중심의 역량 구성을 서둘러야 한다. 속도가 무엇보다 중요하다. 빠른 피드백 루프는 집중력을 유지할 수 있게 도와주며, 더 많은 실험을 가능하게 한다. 캐시를 적극적으로 사용하여 파이프라인 전 단계에서 병목과 대기 시간을 최소화하라.

## 4. 도구
Claude와 Claude Code는 현재 가장 균형 잡힌 성능과 업데이트 속도를 제공하며 경쟁 제품 대비 약 6개월 정도 앞서 있다는 평가를 받는다. 다만 도구 우위는 고정된 사실이 아니므로 분기마다 모델 품질, 편집 경험, 협업 기능, 보안 옵션, 비용 등을 비교하는 평가 루틴을 운영하고, YOLO 모드를 기본값으로 택해 사람의 수동 리뷰보다는 자동화된 테스트·리뷰·백업 루틴으로 신뢰를 확보한다. DevOps 툴체인(CI/CD, IaC, 모니터링 등)과의 긴밀한 통합을 통해 AI가 제안한 변경을 빠르게 배포하고 관찰할 수 있도록 환경을 꾸민다. 팀은 CLI를 선호해 스크립트를 통한 자동화를 쉽게 하고 토큰 사용량을 줄이며, MCP는 조사 목적의 Perplexity MCP와 최신 문서 참조용 Context7 MCP 정도만 최소한으로 사용한다. 언어와 스택 측면에서는 백엔드에 Go, 프론트엔드에 TypeScript/React를 주력으로 삼고 있으며, Go는 빠른 컴파일과 정적 타입, 간결한 문법 덕분에 바이브 코딩에 특히 적합하다는 결론을 내렸다. 멀티플랫폼 개발에는 Flutter/Dart 조합을 사용하고, Makefile로 프로젝트 구조·테스트 스캐폴딩·린터 설정을 템플릿화해 AI에게도 동일한 지침을 제공한다. 새로운 언어나 프레임워크를 도입할 때에는 PoC→파일럿→확산 단계를 밟으면서 프롬프트와 코드 리뷰 체크리스트를 함께 갱신한다. 컨테이너는 안정적인 개발과 배포를 지원하며, 데이터베이스는 목적에 맞는 것을 선택하되 빠른 프로토타이핑이 필요하다면 SQLite를 권장한다.

## 5. 프롬프팅
좋은 프롬프트는 요청뿐만 아니라 숨은 의도를 함께 전달한다. XY 문제처럼 진짜 니즈를 공유하지 않으면 모델은 표면적 해결책만 제시하게 된다. 따라서 “최신 항목을 먼저 보여주려 한다”와 같은 목적을 컨텍스트로 제공하고, 당면한 제약과 성공 기준까지 함께 설명한다. 작업을 시작하기 전에 명확하지 않은 부분이 있다면 질문하도록 명시해 두면, Anthropic 엔지니어가 제안한 방법론(https://x.com/trq212/status/2005315275026260309)처럼 AI가 스스로 요구사항을 확인하는 루프를 만들 수 있다. 반복적으로 사용하는 프롬프트는 Claude Code의 에이전트, 커맨드, 훅, 스킬, 플러그인 기능을 활용해 자산화하고, 저장소에서 버전 관리하여 팀 전원이 동일한 지침을 호출하도록 한다. 해결까지 걸린 턴 수, 프롬프트 재작성 횟수 같은 지표를 수집하면 프롬프트 품질 개선에도 도움이 된다.

## 6. 워크플로우
전체 워크플로우는 요구사항 분석, 테크 스택 및 아키텍처 설계, 계획 수립, 구현·검증·배포 루프로 이어지며 정기적으로 전체 리뷰를 거쳐 대대적인 리팩토링을 수행하는 사이클을 포함한다. 요구사항 단계에서는 사용자 시나리오, 제약, 성공 기준을 문서화해 AI에게도 제공하며, 아키텍처 단계에서는 후보 설계안을 AI와 브레인스토밍한 뒤 핵심 결정을 Decision Log에 남긴다. 계획 수립 시에는 작업을 확인 가능한 단위로 나누고 각 단위의 입력·출력·검증 방법을 정의한다. 실행 루프에서는 프롬프트, 코드 생성, 테스트, 리뷰, 배포의 순서를 짧은 주기로 반복하며, 각 단계 사이의 대기 시간을 자동화로 최소화하고, 루프 몇 회차마다 전체 시스템을 리뷰해 방향을 재조정한다. 비 바이브 코딩 프로젝트에서 바이브 코딩으로 전환하려면 문서화, 자동화 테스트 도입, CI/CD 파이프라인 구축이 반드시 선행되어야 한다.

## 7. 품질 관리
기능을 가능한 작은 단위로 쪼개어 구현과 테스트를 동시에 진행하면 AI가 만들어 내는 결과를 실시간으로 검증할 수 있다. 각 단위에는 최소 하나의 자동화 테스트를 포함시키고, 단위 테스트·스냅샷 테스트·계약 테스트 같은 층위를 조합해 신뢰도를 높인다. 자동화된 테스트와 CI/CD 파이프라인은 Git 혹은 Claude Code의 훅 기능을 이용해 구성하며, UT, 린트, 보안 스캔 등을 촘촘하게 배치해 YOLO 모드에서도 신뢰할 수 있는 가드레일을 유지한다. 모든 변경은 이 파이프라인을 통과해야 하며, 실패했을 때는 해당 프롬프트와 로그를 함께 기록해 다음 반복에서 AI 응답을 개선하는 데이터로 쓴다. 코드 리뷰에서는 보안 취약점, 데이터 흐름, 예외 처리뿐 아니라 프롬프트와 자동화 자산이 재사용 가능한 형태로 남았는지까지 확인한다.

## 8. 보안
AI 도구 사용 시에는 ZDR(Zero Data Retention) 모드를 필수 기본값으로 설정하고, 조직의 데이터 라벨링·마스킹 정책을 준수해야 한다. SAST/DAST 도구와 AI 기반 코드 스캐너를 CI에 통합해 자동화된 보안 리뷰를 파이프라인 속에 포함시키고, 모델 프롬프트에도 보안 정책을 명시하여 정책을 위반하는 코드가 생성되면 자동으로 플래그하도록 룰을 구성한다. 민감한 데이터나 비공개 아키텍처 정보를 공유할 때는 최소한의 문맥만 제공하도록 프롬프트 템플릿을 설계한다.

## 9. 협업
바이브 코딩 시대에는 프롬프트, 프로세스, 자동화가 주요 자산이므로 개인 소유 개념 대신 팀 차원의 관리 체계를 확립한다. 개발자 한 명이 커버할 수 있는 영역이 넓어진 만큼 오너십을 재정의하고, 저장소나 Claude 플러그인을 통해 프롬프트와 프로세스를 공유해 팀 전원이 동일한 세팅을 사용하게 한다. 사내 바이브 코딩 커뮤니티를 만들어 모범 사례와 실패 사례를 신속히 전파하며, 페어 프로그래밍이나 몹 프로그래밍 세션을 활용해 프롬프트 설계와 리뷰를 공동으로 진행하고, 세션 로그를 지식 베이스에 보관해 신입 구성원이 빠르게 온보딩하도록 돕는다.
