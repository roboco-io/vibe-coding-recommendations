## 1. 바이브 코딩이란?

단순한 코드 생성을 넘어서

- 설계, 구현, 검증, 문서화, 운영 전반에 AI 활용
- 문제 정의와 가설 검증 루프 단축이 핵심
- 코드 작성 속도보다 의사결정 속도에 집중

---

## 바이브 코딩의 범위

```mermaid
graph LR
    A[문제 정의] --> B[설계]
    B --> C[구현]
    C --> D[검증]
    D --> E[배포]
    E --> F[피드백]
    F --> A
```

모든 단계에서 AI가 가속화

---

## 2. 핵심 원칙

기존 소프트웨어 개발의 연속선

- SOLID, TDD, 디자인 패턴은 여전히 유효
- AI는 이러한 실천을 더 빠르게 반복하게 하는 촉매
- 모던 소프트웨어 엔지니어링의 가속화

---

## 신뢰할 수 있는 프로세스

> **AI를 믿지 말고, 사람도 믿지 말라!**

- 자동화된 테스트
- 자동화된 코드 리뷰
- 자동화된 백업 루틴

검증 가능한 프로세스가 핵심

---

## 생산성 공식

```
바이브 코딩 생산성 =
  개발자 역량 × AI 도구와 프로세스의 생산성
```

- 곱셈 효과: 역량이 높을수록 AI가 증폭
- 코드 작성 능력보다 문제 해결력이 중요
- 도메인 지식, 아키텍처, 프로세스, 보안 역량

---

## 개발자 역량의 재정의

코더에서 문제 해결자로

- 문제 정의 능력
- 도메인 지식
- 아키텍처 설계
- 프로세스 설계
- 보안 지식

---

## 속도가 무엇보다 중요

**빠른 피드백 루프**

- 집중력 유지와 더 많은 실험
- 파이프라인 전 단계 병목 최소화
- 캐시 적극 활용
- 대기 시간 제거

프로그래밍 언어를 직접 다루지 않는 시대 대비

---

## 3. 도구와 환경

**현재 최선: Claude + Claude Code**

- 경쟁자 대비 6개월 앞선 것으로 평가
- 하지만 분기마다 재평가 필요
- 시장은 빠르게 변화 중

---

## YOLO 모드

과감한 실행 + 탄탄한 안전망

- 사람의 수동 리뷰보다 자동화된 검증
- 빠르게 시도하고 빠르게 롤백
- CI/CD, IaC, 모니터링 필수

---

## 언어 선택

**Go + TypeScript/React**

Go가 바이브 코딩에 적합한 이유:
- 빠른 컴파일 (즉시 검증)
- 정적 타입 (오류 조기 발견)
- 간결한 문법 (AI가 이해하기 쉬움)
- 뛰어난 성능 (프로토타입→프로덕션)

멀티플랫폼: Flutter/Dart

---

## 컨테이너와 데이터베이스

**컨테이너**
- 안정적인 개발과 배포 지원
- 일관된 환경 보장

**데이터베이스**
- 목적에 맞는 선택
- 빠른 프로토타이핑: SQLite 권장
- 프로토타입→프로덕션 전환 용이

---

## CLI vs MCP

**CLI 선호**

- 스크립트 자동화 용이
- 토큰 사용량 적음

**MCP 최소 사용**
- Perplexity MCP (조사)
- Context7 MCP (최신 문서)

## 4. 효과적인 프롬프팅

**의도를 전달하라**

나쁜 예:
```
리스트를 역순으로 정렬해줘
```

좋은 예:
```
사용자에게 최신 항목을 먼저 보여주고 싶어.
현재 리스트가 생성일 오름차순인데,
최선의 방법은 무엇일까?
```

---

## XY 문제

- X: 진짜 해결하려는 문제
- Y: 내가 생각한 해결 방법

**X를 공유하라, Y만 물어보지 말라**

숨은 의도, 제약, 성공 기준을 함께 전달

## AI에게 질문을 허락하라

```
작업을 시작하기 전에
명확하지 않은 부분이 있다면 질문해줘.
```

- AI가 요구사항을 확인하는 루프
- 추측보다 명확화

## 7. 보안

**ZDR 모드 필수**

Zero Data Retention
- 코드가 학습 데이터로 사용되지 않음
- 민감한 프로젝트 필수
- 개인 프로젝트도 권장

---

## 자동화된 보안 리뷰

**파이프라인 통합**

- SAST/DAST 도구
- AI 기반 코드 스캐너
- OWASP Top 10 체크
- 시크릿 감지

프롬프트에 보안 정책 명시

## 8. 팀 협업

**오너십 재정의**

코드 작성자 → 코드 검증자/책임자

- AI가 작성해도 개발자가 책임
- 개인 소유 → 팀 공동 소유
- 넓어진 커버리지, 높아진 일관성 필요
